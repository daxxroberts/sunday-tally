### Service ↔ Ministry composition model — Builder-confirmed 2026-06-02 (for #36; not yet migrated)
**Confirmed mental model:** A *service occurrence* is a CONTAINER. One or more *ministries* run inside it (parallel building blocks). The Builder enters one number per ministry per occurrence; the **service-level total = SUM of the ministries in that occurrence** (e.g. 9:00 Sunday total attendance = Experience 420 + LifeKids 85 = 505). Each ministry can carry its OWN metric set (different reporting tags per ministry is allowed).
- **D-072** No "is-ministry" flag. `service_tags` already IS the ministry axis (table-level identity). Ministry-level reporting is already native via `metrics.ministry_tag_id` — service-independent rollups need no change.
- **D-073** Add `service_template_tags` (service_template_id, ministry_tag_id, sort_order; UNIQUE(template,tag)) — the many-to-many service→ministry composition link. Drives the #36 entry screen (which ministry sections to show). Backfill every existing `service_templates.primary_tag_id` in as a peer row; leave that column physically in place (additive migration 0028; no destructive drop on locked schema). **No `is_primary` — see D-076.**
- **D-074** Service-occurrence total is **DERIVED (SUM across constituent ministries), NEVER STORED** — same principle as the "totals always calculated" critical rule. Editor records per-ministry atoms (ministry × reporting_tag × instance → metric_entry); the total is a view/query. **Rollup is keyed on `service_instance_id`** (group by occurrence → one total per occurrence). Composition link is NOT needed for the total — total just sums whatever entries share the occurrence id; the link only drives which empty input sections the entry screen renders.
- **D-075** Composition is **template-level** (stable per recurring service), not per-occurrence — Builder-confirmed 2026-06-02. A ministry that doesn't meet a given week (holiday) is handled via `metric_entries.is_not_applicable`, not by re-tagging the occurrence.
- **D-076** Ministries within a service are **equal peers — no controlling/primary ministry** (Builder-confirmed 2026-06-02). Rollup groups by occurrence id; neither ministry is parent of the rollup. `is_primary` dropped as a concept; only `sort_order` remains (pure display ordering, zero logic). A cosmetic "defining service" label can be added later if ever needed — nothing depends on it now.
- **D-078** **Entry flow = ministry-first, two-zone hybrid** (Builder-confirmed 2026-06-02). Top zone: per-occurrence, ministry-one-at-a-time, **instance-scoped** metrics (attendance, volunteers, response stats) → `metric_entries.service_instance_id`. Each ministry shows only its own metric set. Bottom zone: a **once-a-week "This Week" step** for **period-scoped** metrics → `metric_entries.period_anchor` (the week's Sunday), church-wide (`OTHER`/CHURCH_WIDE ministry per D-068), entered once regardless of occurrence count. **Giving for this church is period/weekly** (one number covering all services that week — no per-service giving). `metrics.scope` ('instance'|'period') is per-metric/per-church, so a church that tracks giving per service configures it instance-scoped instead — schema supports both (D-064 XOR was built for exactly this). Proposed UX: ministry-driven flow (pick occurrence → enter each ministry) then a closing "This Week" panel for period metrics. **Reaffirmed 2026-06-02** after reconsidering metric-first (enter all attendance, then all volunteers): staying ministry-first. NOTE: entry layout is a free UI choice — metric-first writes identical metric_entries, so a metric-first view or a both-with-toggle can be added later with zero schema impact if ever wanted. **OPEN (goal E, session model): SUNDAY_SESSION is keyed per-DATE (`sunday_session_[YYYY-MM-DD]`) but period metrics anchor per-WEEK — the session must anchor period-scoped entry at the week even though occurrences are per-day.**

(a) **Volunteers expand into areas/roles with a live calculated subtotal** — each area is its own VOLUNTEERS-tagged metric; ministry volunteer total = SUM, never stored (rule #3); matches imported tall-format data.
